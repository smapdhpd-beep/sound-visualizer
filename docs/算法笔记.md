# 算法笔记

## 1. Chladni 物理公式演进

### v1.0：多波源行波干涉（已废弃）

```
val = abs(sum(sin(2*pi*fi * (t - di/v))))
force = -val * grad(val)
```

**问题**：
- 4 个固定波源产生的干涉图案细碎，sym=1 时像散点
- 频率映射方向易反（bass→freq 导致低音产生高频图案）
- 波源旋转/振荡导致节线漂移，画面模糊

**废弃原因**：无法在不使用高对称复制的情况下产生清晰可辨的大尺度几何图案。

---

### v1.1：极坐标驻波（当前）

```
r = dist(x,y,center) / maxR
amp = abs(sin(m * pi * r) * cos(n * theta))
```

**物理意义**：
- `sin(m*pi*r)`：径向驻波，m 个半波长在 [0,1] 半径内
- `cos(n*theta)`：角向驻波，n 个周期在 0~2pi 内
- 波节线：`amp=0` 的位置，即 `sin(m*pi*r)=0` 或 `cos(n*theta)=0`
- 自然对称阶数：`2n`（cos 的波节线每 pi/n 一个）

**优势**：
- m/n 为整数时产生稳定、可预测的几何图案
- m=2,n=1 → 左右对称的半圆环
- m=4,n=3 → 12 瓣星形
- 不需要 symmetry 参数也能看到清晰图案（因为 `2n` 自带对称）

## 2. 节点粘性算法

**背景**：粒子被梯度力推向波节线后，布朗运动（恒定 45）立即将其踢走，导致堆积密度极低。

**算法**：
```
nodeProx = 1 - min(1, amp)
if nodeProx > 0.55:
    stick = (nodeProx - 0.55) / 0.45 * 0.85
    effBrownian *= (1 - stick)   // 最多降 85%
    effDamp = 1 - (1-damp)*(1-stick*0.5)  // 阻尼最多升 42.5%
```

**效果**：
- sym=1 平均亮度从 49→140（+186%）
- bright 像素比例从 26%→53%

**关键参数**：
- 阈值 0.55：太深则粘性范围窄，太浅则粘性太弱
- stick 系数 0.85：最大粘性程度
- damp 增益 0.5：粘性区阻尼增强的倍率

## 3. FFT 峰值 → m/n 映射

**峰值检测**：
```
_findPeaks(): 找频谱局部最大值（5 点邻域），取前 3
```

**对数映射**（解决低频 bin 范围小的问题）：
```
logFreq = log2(bin + 1)
norm = logFreq / log2(binCount)
targetM = 1 + round(norm * 9)  // 1~10
```

**频带能量修正**：
- bassRatio > 50% → 限制 n=1~2, m=2~5（大尺度圆环）
- highRatio > 40% → 允许 m,n=10（细密星形）

**为什么用对数映射**：
- 音乐基频通常在 200~1000Hz（bin 8~43，sampleRate=48k, fft=2048）
- 线性映射会将大部分音乐压缩到 m=1~2
- 对数映射让 200Hz→m≈3，1000Hz→m≈6，覆盖更均匀

## 4. 离线验证的局限性

**差异维度**：

| 维度 | node-canvas (Cairo) | 浏览器 Canvas 2D |
|------|---------------------|------------------|
| 合成后端 | CPU 软件合成 | GPU 加速 (Skia/CoreGraphics) |
| 颜色空间 | 线性数学运算 | sRGB + 浏览器色彩管理 |
| DPR | 无 (1:1) | 按 devicePixelRatio 缩放 (Retina 2x/3x) |
| 视觉感知 | 单帧快照 | 连续视觉暂留 + 拖尾流动感 |
| 音频输入 | 硬编码数值 | 真实频谱分布 + 底噪 |

**结论**：
- 离线测试只能验证"算法逻辑是否正确"
- 最终视觉效果必须以浏览器实机为准
- 量化分析（亮度统计）比肉眼猜测可靠，但仍不能替代实机

## 5. 频率平滑层级

```
原始信号 → AudioEngine EMA (a=0.75) → 渲染器二次平滑 (lerp=0.08~0.15)
```

**为什么需要两层**：
- AudioEngine 的 EMA 防止音频跳变，但仍保留音乐动态
- 渲染器的二次平滑让 m/n 模态缓慢过渡，防止节线抖动
- 每层平滑系数不同：特征层 0.75（较快），物理层 0.08~0.15（较慢）

**原则**：视觉稳定比"实时响应"更重要。人眼对 0.5 秒内的模态跳变不敏感，但对每帧抖动极其敏感。

## 6. NaN 瘟疫：从空数组到画面全黑（2026-06-16）

### 故障现象

歌曲循环播放（replay）时，画面突然全黑，且永久无法恢复。刷新页面后正常，但再次循环又黑屏。

### 根因链

```
音频循环点 → analyser 读全 0
  → _findPeaks() 无峰值 → 返回 []（空数组）
  → peakBins = []，pb[0] = undefined
  → Math.log2(undefined + 1) = NaN
  → targetM = NaN
  → smoothedM += (NaN - smoothedM) * 0.15 → smoothedM 永久 NaN
  → _waveField: Math.sin(NaN * PI * r) = NaN
  → push = -NaN * strength = NaN
  → particle.x = NaN，所有粒子移出画布
```

### 为什么空数组能穿透防御？

```javascript
const pb = peakBins || [10, 30, 50]; // 错误！[] 不是 falsy
// peakBins = [] 时，pb = []，pb[0] = undefined

const pb = (peakBins && peakBins.length >= 2) ? peakBins : [10, 30, 50]; // 正确
```

### 为什么 NaN 是瘟疫？

| 运算 | 结果 | 含义 |
|------|------|------|
| `NaN + 5` | `NaN` | 加法传播 |
| `NaN * 0` | `NaN` | 连乘零都救不了 |
| `NaN === NaN` | `false` | 无法通过 `===` 检测 |
| `isNaN(NaN)` | `true` | 唯一检测方式 |
| `Math.round(NaN)` | `NaN` | 取整传播 |
| `Math.sin(NaN)` | `NaN` | 三角函数传播 |

**一旦状态变量被 NaN 污染，永远无法通过正常运算自愈。**

### 三层防御体系

**上游防御（数据源）**：
```javascript
_findPeaks() {
  const result = peaks.slice(0, 3);
  while (result.length < 3) {
    result.push({ bin: [10,30,50][result.length], value: 0 });
  }
  return result;
}
```

**中游防御（输入校验）**：
```javascript
const safeBin = (b) => (typeof b === 'number' && !isNaN(b) && b > 0) ? b : 10;
const bin1 = safeBin(pb[0]);
```

**下游防御（状态自检）**：
```javascript
this.smoothedM += (targetM - this.smoothedM) * responseSpeed;
if (isNaN(this.smoothedM)) this.smoothedM = 3; // 自愈重置
```

### 调试方法

如果画面突然全黑且无法恢复：
1. 检查 `renderer.smoothedM` 是否为 NaN
2. 检查 `features.peakBins` 是否为空数组
3. 在 `update()` 开头添加 `console.assert(!isNaN(this.smoothedM), 'smoothedM NaN!')`

## 7. Canvas DPR 矩阵保护（2026-06-16）

### 故障现象
切换歌曲后调用 `renderer.reset()`，画面出现"画中画"：旧画面残留为背景，新画面变成一个贴在左上角的小长方形。

### 根因
App 在 `_resize()` 中设置了 DPR 缩放矩阵：
```javascript
const dpr = Math.min(window.devicePixelRatio, 2);
this.ctx2d.setTransform(dpr, 0, 0, dpr, 0, 0);
```

但 `ChladniRenderer.reset()` 中为了"清屏"擅自重置了变换矩阵：
```javascript
this.ctx.setTransform(1, 0, 0, 1, 0, 0); // 破坏 DPR
this.ctx.fillRect(0, 0, this.width, this.height); // 只画了左上角 1/dpr² 区域
```

之后 `render()` 每帧的拖尾填充也只在逻辑像素范围内操作，导致：
- 旧画面其余区域永远不被刷新（残留）
- 粒子坐标在无缩放坐标系下绘制，全部挤在左上角

### 修复
**Renderer 不应修改 `setTransform`**，除非立即恢复。清屏应使用 App 已设置好的当前坐标系：
```javascript
// 正确：保持当前 transform（含 DPR），fillRect 逻辑尺寸正好覆盖全屏
this.ctx.fillRect(0, 0, this.width, this.height);
```

或参照 `_clearCanvas2D()` 的模式：保存→重置→清屏→恢复。

### 原则
- **谁设置，谁管理**：DPR 矩阵是 App 层级的全局状态，Renderer 只读取不修改
- **Renderer 的 `this.width/height` 始终是逻辑尺寸**，配合 App 的 DPR transform 正好覆盖物理像素

## 8. 颜色模式设计：迷幻但不脏（2026-06-16）

### 需求矛盾
- "颜色随机变化" → 容易联想到彩虹/多色粒子 → 画面混沌、对比度下降、图案被颜色淹没
- "不能脏" → 需要保持高对比度和清晰的图案结构

### 方案对比

| 方案 | 实现 | 干净度 | 迷幻感 | 结论 |
|------|------|--------|--------|------|
| 完全随机 RGB | 每帧随机 theme.particle | 极脏 | 闪烁 | 废弃 |
| 多色粒子 | 粒子按位置映射不同 hue | 脏 | 强 | 废弃 |
| **HSL 单色调漫游** | **所有粒子共享同一 hue，饱和度固定，只变亮度** | **极干净** | **强** | **采用** |

### 为什么单色调漫游最干净？

人眼对"色相差异"极其敏感，多色并列会立即产生"喧闹"感；但对"同一色相的亮度差异"不敏感，视为"同一物体的明暗"。

因此：
- **色相（H）**：全局共享，随时间匀速旋转 + onset 跳跃 → 产生迷幻流转感
- **饱和度（S）**：固定 85% → 始终保持鲜艳，不灰不脏
- **亮度（L）**：60%~85%，由粒子堆积度 `acc` 控制 → 波节线更亮，非节线更暗，图案结构完全保留

### 音频映射
```javascript
// 基础旋转：20°/s + 整体能量加速（最高 80°/s）
this.hue += (20 + overall * 60) * dt;

// Onset 跳色：鼓点触发时瞬间跳跃 40~80°
if (onset > 0.3) this.hue += 40 + Math.random() * 40;
```

### 架构实现
- App 层：`globalParams.theme` 增加 `'audioDriven'` 选项
- `_resolveTheme()`：audioDriven 时返回 `{ mode: 'audioDriven', bg: [0,0,0], particle: [255,255,255] }`
- Renderer 层：`render()` 中根据 `theme.mode === 'audioDriven'` 切换 `hsla()` / `rgba()`
- 其他 renderer 不受影响（它们只读取 `theme.particle`，不关心 `mode`）

## 9. 频谱瀑布：像素级滚动与透视变形（2026-06-17）

### 为什么用离屏 Canvas 像素滚动？

频谱瀑布需要维护约 120 帧历史 × 128 频带。如果每帧用 `fillRect` 重绘 1.5 万个矩形，性能勉强及格但发热明显。改用 **离屏 Canvas + putImageData + drawImage 自滚动** 后：
- 每帧只需：1 次 `drawImage`（整图下移 1px）+ 1 次 `putImageData`（顶行写入）+ 1 次 `drawImage`（主画布贴图）
- 复杂度从 O(bands×history) 降至 O(bands + history)

### 数据结构

```
离屏 canvas: width=bandCount(128), height=historyLength(120)
每帧 update:
  1. 把离屏 canvas [0, 0, 128, 119] 画到 [0, 1, 128, 119]（下移 1px）
  2. 用 ImageData 生成新频谱行，putImageData 到 (0,0)
每帧 render:
  平面模式: drawImage(离屏, 主画布全屏)
  透视模式: 逐行 drawImage，每行独立计算宽度 scale
```

### 对数频率映射

人耳感知频率是对数的。AudioEngine 降采样时使用幂律分布：
```javascript
start = pow(i / bands, 1.6) * binCount
end   = pow((i+1) / bands, 1.6) * binCount
```
指数 1.6 让低频区获得更多视觉频带，高频区压缩，与听觉感知一致。

### 透视变形算法

```javascript
progress = row / history;          // 0=顶部(新), 1=底部(旧)
scale    = 1 - progress * perspective * 0.6;
drawW    = width * scale;
drawX    = centerX - drawW / 2;
```

逐行 `drawImage(src, row)` 贴到梯形对应位置，形成"从远处看瀑布"的 3D 感。

### 与 onset 的联动

`onsetFlash` 采用指数衰减而非线性：
```javascript
if (onset > 0.25) this.onsetFlash = 1.0;
this.onsetFlash *= pow(0.02, dt); // 约 50ms 内从 1.0 衰减到接近 0
```
这样鼓点触发时全屏极短暂微闪（不遮挡频谱），像相机的闪光灯，节奏感强但不刺眼。

## 10. 克拉尼图多形态边界（2026-06-17）

### 统一架构

所有边界共享同一套粒子系统、节点粘性算法、音频映射逻辑。只有 `_waveField` 和 `_applyBoundaryPull` 按形态分发：

```javascript
_waveField(px, py, m, n, time, complexity) {
  switch (this.params.boundaryShape) {
    case 'square':  return this._waveFieldSquare(...);
    case 'annular': return this._waveFieldAnnular(...);
    case 'ellipse': return this._waveFieldEllipse(...);
    default:        return this._waveFieldCircle(...);
  }
}
```

### 圆形（Circle）— 基准

```
r = dist / maxR
amp = abs(sin(m·π·r) · cos(n·θ))
```
- 径向驻波 × 角向驻波
- 波节线：同心圆 + 放射线
- 自然对称阶数：2n

### 方形（Square）— 笛卡尔坐标

```
u = (x/maxR + 1) * 0.5   // [0, 1]
v = (y/maxR + 1) * 0.5
amp = abs(sin(m·π·u) · sin(n·π·v)) · fade
```
- 两个正交驻波的乘积
- 波节线：水平线 + 垂直线 → 网格交叉
- m=2,n=2 → 四宫格；m=3,n=2 → 3×2 网格
- 边界用 max(|nx|,|ny|) 做软衰减，模拟固定边界膜

### 环形（Annular）— 带内孔

```
rNorm = (r - innerR) / (1 - innerR)   // [0, 1], innerR=0.35
amp = abs(sin(m·π·rNorm) · cos(n·θ))
```
- 内孔是禁区：r < innerR 时 amp=1（波腹，粒子被推开）
- 外圈软边界 + 势垒：r > 1.2 时拉回
- 视觉上像靶心、瞳孔、星际环
- 内孔附近增加 barrier 项，防止粒子卡入空洞

### 椭圆（Ellipse）— 变形极坐标近似

```
rEff = sqrt((dx/a)² + (dy/b)²) / maxR   // a=1.0, b=0.65
θEff = atan2(dy/b, dx/a)
amp = abs(sin(m·π·rEff) · cos(n·θEff))
    + 0.35·sin((m+1)·π·rEff)·cos((n+1)·θEff·2)   // 二阶猫眼调制
```
- 不是严格马丢函数解（避免引入数学库）
- 变形比例 b=0.65 产生明显椭圆拉伸
- 二阶谐波在 complexity≥1 时启用，产生"猫眼/四叶"独特图案
- 视觉语言独一无二：变形的对称

### 边界拉回策略对比

| 形态 | 拉回逻辑 | 特殊处理 |
|------|----------|----------|
| 圆形 | 超出 maxR×1.2 时径向拉回 | 无 |
| 方形 | x/y 独立检查，超出 ±1.2 时轴向拉回 | 无 |
| 环形 | 外圈径向拉回 + **内孔反向推离** | 内孔禁区防止粒子坠入空洞 |
| 椭圆 | 超出有效半径时沿变形坐标拉回 | 无 |

### 参数配置

Tweakpane 新增「边界形状」下拉：圆形 / 方形 / 环形 / 椭圆。切换时粒子不重置，逐渐漂移到新节线，过渡柔和。
